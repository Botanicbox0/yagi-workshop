## ADR-001: Source of truth = `.md`; `.ts` is generated
Date: 2026-04-23
Status: Accepted

### Context
Design system definitions had two potential habitats: `.md` specs (human-readable, rationale-carrying) and `.ts` tokens (machine-consumable, imported by app code). Keeping both hand-editable causes drift. A linter catches literal mismatches but not semantic ones, and splits reviewer attention across two files per change.

### Decision
`.md` files under `.yagi-autobuild/design-system/` are the sole source of truth. `.ts` in `src/design-tokens/*` is generated by `skills/token-sync.md`. Generated files carry the ARCH §4.2 header marker; pre-commit hook rejects `.ts` without it.

### Consequences
- Positive: drift eliminated by construction; reviewers focus on one file per change; decisions travel with their reasoning.
- Negative: token changes require a manual sync step until CI automation lands (ARCH §4.5); `.ts` diffs are noisy in PRs until scoped regeneration exists.
- Followups: promote `pnpm tokens:verify` to CI check after Phase 3.0 (ARCH §4.5 open).

### Alternatives considered
- Both editable with drift-lint — rejected: linters don't catch semantic drift.
- `.ts` as source, `.md` generated from JSDoc — rejected: JSDoc can't carry per-locale rationale or ADR cross-refs.
- External tool (Figma Tokens) as source — deferred to ARCH §16.3.

## ADR-002: Pretendard Variable as single font for KR + EN
Date: 2026-04-23
Status: Accepted

### Context
YAGI Workshop runs KR primary, EN secondary. Industry standard pairs a Korean-first face with a Latin-first face (Pretendard + Inter; Noto Sans KR + SF Pro). Pretendard Variable already embeds Latin glyphs derived from Inter, tuned to the Korean baseline. The gap between "Pretendard Latin" and standalone Inter is visually small at product-UI sizes (12–32px) and negligible in Korean-dominant layouts.

### Decision
`Pretendard Variable` is the sole custom font. Fallback chain: `"Pretendard Variable", Pretendard, -apple-system, BlinkMacSystemFont, "Apple SD Gothic Neo", "Noto Sans KR", "Malgun Gothic", sans-serif`. Per-scope overrides allowed for editorial surfaces only; must be annotated in `TYPOGRAPHY_SPEC.md §exceptions`.

### Consequences
- Positive: ~40–80KB saved per pageview; KR/EN mixed runs align at baseline; single font-loading path.
- Negative: extreme editorial Latin voices unavailable without scoped exception; Latin-dominant market expansion would likely supersede this ADR.

### Alternatives considered
- Pretendard + Inter — rejected: Inter is already the Latin cut of Pretendard.
- Noto Sans KR + SF Pro — rejected: SF Pro licensing weak off-Apple.
- Per-locale subsetting — rejected: premature optimization.

## ADR-003: Delete `frame-picker.ts`; replace with skill
Date: 2026-04-23
Status: Accepted

### Context
`frame-picker.ts` encoded thumbnail/hero-still frame selection as hardcoded rules (brightness, motion delta, face detection). This is judgment-heavy editorial work — the right frame for KART ZERO differs from FIXERS differs from a digital-human portfolio reel. Rule count grew (10 → 14 → 19); the code became a de-facto spec, and every aesthetic direction change required an engineering PR.

### Decision
Delete `src/lib/design/frame-picker.ts`. Move judgment to `.yagi-autobuild/skills/frame-selection.md` — prose heuristics, anti-patterns, decision trees. Consumed by a human operator or an LLM invoked with the skill attached. If future automation is needed, build an executor agent that reads the skill — do not reintroduce the hardcoded rules.

### Consequences
- Positive: policy is versioned prose, PR-reviewable; creative judgment unblocks from engineering cycles; edge cases refine the skill instead of accumulating `// HACK:` branches.
- Negative: deterministic automation is lost until an LLM-backed executor replaces it; contributors without the skill file get confused.
- Followups: write `skills/frame-selection.md` before any phase that needs frame selection.

### Alternatives considered
- Extract rules into JSON — rejected: still can't express editorial judgment.
- ML classifier — rejected: no labeled data, problem under-specified.

## ADR-004: Modular typographic scale 1.125 (Major Second)
Date: 2026-04-23
Status: Accepted

### Context
Candidate ratios: 1.25 (Major Third, editorial) / 1.2 (Minor Third, general web) / 1.125 (Major Second, product UI). Our surface is dominated by dense product UI — client portals, job queues, dashboards — with occasional editorial moments. Editorial-scale h1 crowds product layouts.

### Decision
Base scale 1.125 at 16px root. Roles: xs 12 / sm 14 / base 16 / lg 18 / xl 20 / 2xl 24 / 3xl 30 / 4xl 36. Editorial heroes may break upward per-scope, annotated at scope root.

### Consequences
- Positive: correct density on dashboards; KR text doesn't blow containers at large sizes; heading hierarchy readable without theatrical.
- Negative: landing pages want larger heroes — scoped override introduces a second mental model; if editorial surface grows in importance, pair with an "editorial scale" ADR.

### Alternatives considered
- 1.25 Major Third — too loud at h1 for product UI.
- 1.2 Minor Third — close runner-up; difference only meaningful above 3xl where scoped overrides already apply.
- Non-modular (hand-picked) — rejected: predictability > per-level aesthetic freedom.

## ADR-005: Expedited Phase Protocol for 1–2 day sprints
Date: 2026-04-23
Status: Accepted

### Context
ARCHITECTURE §5 defines a 6-gate pipeline (CEO → Design Consultation → Plan Design Review → Engineering → Build → Design Review → QA Smoke). ARCHITECTURE §17 forbids skipping gates without an ADR.

Two categories of work do not fit the full pipeline:

1. **Operational tail** — Phase 2.1 style: G1 DNS verify, H1 investigation, seed migrations, DEFER_TO_2_1 triage. No new UI, no new architectural decisions.
2. **MVP sprints** — Phase 2.5 style: ship a minimal feature in 1 day to validate market fit, using only existing design-system primitives.

Running both through all 6 gates would take 5+ days each. Additionally, `reviews/PLAN_DESIGN_REVIEW.md`, `reviews/ENGINEERING_REVIEW.md`, and `skills/office-hours.md` / `design-consultation.md` are not yet written — a full-pipeline phase currently cannot execute as specified.

### Decision
Introduce **Expedited Phase** as a formal mode. A phase qualifies as Expedited if **all** of the following hold:

1. Duration ≤ 2 working days.
2. CEO (야기) is directly in the loop for all gates that run.
3. No new design tokens, primitives, or patterns — everything reuses existing `design-system/` artifacts.
4. No cross-phase schema changes beyond what's already planned in carryover.
5. Scope fits on one SPEC page (≤ 300 lines).

**Gates that run in Expedited mode**:
- **Gate 1 (CEO)** — emits `gates/phase-{N}/CEO_APPROVED.md`. Uses **pre-filled pattern**: Builder drafts the SPEC + decisions, CEO responds "APPROVED" or "EDIT §N: ...". Absence of objection = approval.
- **Gate 4 (Engineering Review, K-05 Codex)** — emits `gates/phase-{N}/ENGINEERING_APPROVED.md`. Non-negotiable; K-05 is the adversarial check.
- **Build** — atomic commits (Phase 2.0 rule retained).
- **Gate 6 (QA Smoke)** — emits `gates/phase-{N}/QA_SMOKE.md`. Browser smoke test at minimum; Playwright if cost-free.

**Gates skipped in Expedited mode**:
- Gate 2 (Design Consultation) — no new design, nothing to consult.
- Gate 3 (Plan Design Review) — would check new design against rules; no new design.
- Gate 5 (Design Review, post-build) — same reason; skipped unless a Build step introduces a new variant (see triggers below).

**Expedited-forbidden triggers** — if any of these fire mid-phase, the phase halts and is re-classified as a full phase:

- A PR introduces a component variant not in `COMPONENT_CONTRACTS.md`.
- A PR hardcodes a color, size, or radius value.
- A new screen uses a frame not cataloged in `UI_FRAMES.md`.
- Codex K-05 returns a HIGH finding that requires design rework.

### Consequences
- Positive: operational carryover ships in 1 day instead of stalling on empty review files; first-feature MVPs (Challenge, etc.) get fast validation; K-05 retained as the quality floor.
- Negative: UI quality risk — if an Expedited phase accidentally needs design decisions, the forbidden-trigger halt catches it late; some design drift possible if triggers miss an edge case.
- Followups: write `reviews/PLAN_DESIGN_REVIEW.md`, `reviews/ENGINEERING_REVIEW.md`, `skills/office-hours.md`, `skills/design-consultation.md` in a backfill phase (Phase 2.6 candidate). Then promote Phase 3.0+ to full-gate mandatory.

### Alternatives considered
- Run all phases through full 6 gates — rejected: impossible with current skill/review file gaps; 1-day scope infeasible.
- Skip all gates for "emergency" phases — rejected: no quality floor, violates ARCH §17 spirit.
- Write the missing skills/reviews first, then do operational tail — rejected: blocks operational work 3–5 days for no operational benefit.
