## ADR-001: Source of truth = `.md`; `.ts` is generated
Date: 2026-04-23
Status: Accepted

### Context
Design system definitions had two potential habitats: `.md` specs (human-readable, rationale-carrying) and `.ts` tokens (machine-consumable, imported by app code). Keeping both hand-editable causes drift. A linter catches literal mismatches but not semantic ones, and splits reviewer attention across two files per change.

### Decision
`.md` files under `.yagi-autobuild/design-system/` are the sole source of truth. `.ts` in `src/design-tokens/*` is generated by `skills/token-sync.md`. Generated files carry the ARCH §4.2 header marker; pre-commit hook rejects `.ts` without it.

### Consequences
- Positive: drift eliminated by construction; reviewers focus on one file per change; decisions travel with their reasoning.
- Negative: token changes require a manual sync step until CI automation lands (ARCH §4.5); `.ts` diffs are noisy in PRs until scoped regeneration exists.
- Followups: promote `pnpm tokens:verify` to CI check after Phase 3.0 (ARCH §4.5 open).

### Alternatives considered
- Both editable with drift-lint — rejected: linters don't catch semantic drift.
- `.ts` as source, `.md` generated from JSDoc — rejected: JSDoc can't carry per-locale rationale or ADR cross-refs.
- External tool (Figma Tokens) as source — deferred to ARCH §16.3.

## ADR-002: Pretendard Variable as single font for KR + EN
Date: 2026-04-23
Status: Accepted

### Context
YAGI Workshop runs KR primary, EN secondary. Industry standard pairs a Korean-first face with a Latin-first face (Pretendard + Inter; Noto Sans KR + SF Pro). Pretendard Variable already embeds Latin glyphs derived from Inter, tuned to the Korean baseline. The gap between "Pretendard Latin" and standalone Inter is visually small at product-UI sizes (12–32px) and negligible in Korean-dominant layouts.

### Decision
`Pretendard Variable` is the sole custom font. Fallback chain: `"Pretendard Variable", Pretendard, -apple-system, BlinkMacSystemFont, "Apple SD Gothic Neo", "Noto Sans KR", "Malgun Gothic", sans-serif`. Per-scope overrides allowed for editorial surfaces only; must be annotated in `TYPOGRAPHY_SPEC.md §exceptions`.

### Consequences
- Positive: ~40–80KB saved per pageview; KR/EN mixed runs align at baseline; single font-loading path.
- Negative: extreme editorial Latin voices unavailable without scoped exception; Latin-dominant market expansion would likely supersede this ADR.

### Alternatives considered
- Pretendard + Inter — rejected: Inter is already the Latin cut of Pretendard.
- Noto Sans KR + SF Pro — rejected: SF Pro licensing weak off-Apple.
- Per-locale subsetting — rejected: premature optimization.

## ADR-003: Delete `frame-picker.ts`; replace with skill
Date: 2026-04-23
Status: Accepted

### Context
`frame-picker.ts` encoded thumbnail/hero-still frame selection as hardcoded rules (brightness, motion delta, face detection). This is judgment-heavy editorial work — the right frame for KART ZERO differs from FIXERS differs from a digital-human portfolio reel. Rule count grew (10 → 14 → 19); the code became a de-facto spec, and every aesthetic direction change required an engineering PR.

### Decision
Delete `src/lib/design/frame-picker.ts`. Move judgment to `.yagi-autobuild/skills/frame-selection.md` — prose heuristics, anti-patterns, decision trees. Consumed by a human operator or an LLM invoked with the skill attached. If future automation is needed, build an executor agent that reads the skill — do not reintroduce the hardcoded rules.

### Consequences
- Positive: policy is versioned prose, PR-reviewable; creative judgment unblocks from engineering cycles; edge cases refine the skill instead of accumulating `// HACK:` branches.
- Negative: deterministic automation is lost until an LLM-backed executor replaces it; contributors without the skill file get confused.
- Followups: write `skills/frame-selection.md` before any phase that needs frame selection.

### Alternatives considered
- Extract rules into JSON — rejected: still can't express editorial judgment.
- ML classifier — rejected: no labeled data, problem under-specified.

## ADR-004: Modular typographic scale 1.125 (Major Second)
Date: 2026-04-23
Status: Accepted

### Context
Candidate ratios: 1.25 (Major Third, editorial) / 1.2 (Minor Third, general web) / 1.125 (Major Second, product UI). Our surface is dominated by dense product UI — client portals, job queues, dashboards — with occasional editorial moments. Editorial-scale h1 crowds product layouts.

### Decision
Base scale 1.125 at 16px root. Roles: xs 12 / sm 14 / base 16 / lg 18 / xl 20 / 2xl 24 / 3xl 30 / 4xl 36. Editorial heroes may break upward per-scope, annotated at scope root.

### Consequences
- Positive: correct density on dashboards; KR text doesn't blow containers at large sizes; heading hierarchy readable without theatrical.
- Negative: landing pages want larger heroes — scoped override introduces a second mental model; if editorial surface grows in importance, pair with an "editorial scale" ADR.

### Alternatives considered
- 1.25 Major Third — too loud at h1 for product UI.
- 1.2 Minor Third — close runner-up; difference only meaningful above 3xl where scoped overrides already apply.
- Non-modular (hand-picked) — rejected: predictability > per-level aesthetic freedom.

## ADR-005: Expedited Phase Protocol for 1–2 day sprints
Date: 2026-04-23
Status: Accepted

### Context
ARCHITECTURE §5 defines a 6-gate pipeline (CEO → Design Consultation → Plan Design Review → Engineering → Build → Design Review → QA Smoke). ARCHITECTURE §17 forbids skipping gates without an ADR.

Two categories of work do not fit the full pipeline:

1. **Operational tail** — Phase 2.1 style: G1 DNS verify, H1 investigation, seed migrations, DEFER_TO_2_1 triage. No new UI, no new architectural decisions.
2. **MVP sprints** — Phase 2.5 style: ship a minimal feature in 1 day to validate market fit, using only existing design-system primitives.

Running both through all 6 gates would take 5+ days each. Additionally, `reviews/PLAN_DESIGN_REVIEW.md`, `reviews/ENGINEERING_REVIEW.md`, and `skills/office-hours.md` / `design-consultation.md` are not yet written — a full-pipeline phase currently cannot execute as specified.

### Decision
Introduce **Expedited Phase** as a formal mode. A phase qualifies as Expedited if **all** of the following hold:

1. Duration ≤ 2 working days.
2. CEO (야기) is directly in the loop for all gates that run.
3. No new design tokens, primitives, or patterns — everything reuses existing `design-system/` artifacts.
4. No cross-phase schema changes beyond what's already planned in carryover.
5. Scope fits on one SPEC page (≤ 300 lines).

**Gates that run in Expedited mode**:
- **Gate 1 (CEO)** — emits `gates/phase-{N}/CEO_APPROVED.md`. Uses **pre-filled pattern**: Builder drafts the SPEC + decisions, CEO responds "APPROVED" or "EDIT §N: ...". Absence of objection = approval.
- **Gate 4 (Engineering Review, K-05 Codex)** — emits `gates/phase-{N}/ENGINEERING_APPROVED.md`. Non-negotiable; K-05 is the adversarial check.
- **Build** — atomic commits (Phase 2.0 rule retained).
- **Gate 6 (QA Smoke)** — emits `gates/phase-{N}/QA_SMOKE.md`. Browser smoke test at minimum; Playwright if cost-free.

**Gates skipped in Expedited mode**:
- Gate 2 (Design Consultation) — no new design, nothing to consult.
- Gate 3 (Plan Design Review) — would check new design against rules; no new design.
- Gate 5 (Design Review, post-build) — same reason; skipped unless a Build step introduces a new variant (see triggers below).

**Expedited-forbidden triggers** — if any of these fire mid-phase, the phase halts and is re-classified as a full phase:

- A PR introduces a component variant not in `COMPONENT_CONTRACTS.md`.
- A PR hardcodes a color, size, or radius value.
- A new screen uses a frame not cataloged in `UI_FRAMES.md`.
- Codex K-05 returns a HIGH finding that requires design rework.

### Consequences
- Positive: operational carryover ships in 1 day instead of stalling on empty review files; first-feature MVPs (Challenge, etc.) get fast validation; K-05 retained as the quality floor.
- Negative: UI quality risk — if an Expedited phase accidentally needs design decisions, the forbidden-trigger halt catches it late; some design drift possible if triggers miss an edge case.
- Followups: write `reviews/PLAN_DESIGN_REVIEW.md`, `reviews/ENGINEERING_REVIEW.md`, `skills/office-hours.md`, `skills/design-consultation.md` in a backfill phase (Phase 2.6 candidate). Then promote Phase 3.0+ to full-gate mandatory.

### Alternatives considered
- Run all phases through full 6 gates — rejected: impossible with current skill/review file gaps; 1-day scope infeasible.
- Skip all gates for "emergency" phases — rejected: no quality floor, violates ARCH §17 spirit.
- Write the missing skills/reviews first, then do operational tail — rejected: blocks operational work 3–5 days for no operational benefit.

## ADR-006: SPEC-to-kickoff alignment protocol
Date: 2026-04-23
Status: Accepted

### Context
On Phase 2.1 kickoff, the chat-level prompt to Builder diverged from the committed `SPEC.md` + `CEO_APPROVED.md` in several places: task count (kickoff "5 tasks" vs SPEC 8 groups), scope subset (kickoff omitted G3 yagi-internal seed + G4 POPBILL guard), group labels (kickoff "G1" covered SPEC G1+G2+G5+G6 partial), and triage item count (kickoff "7 items" vs SPEC "15" vs actual `G4_TRIAGE.md` 24). The drift was caught pre-execution by Builder flagging the misalignments explicitly and the user retracting the kickoff in favor of SPEC. Had it not been caught, Phase 2.1 would have shipped missing two SPEC success criteria (G3 seed + G4 POPBILL guard).

Kickoffs are authored in chat flow, often from memory of the SPEC conversation rather than with SPEC.md open. That's fast but lossy — especially when SPEC has 150+ lines, 8 groups, and 7 success criteria.

### Decision
Post-approval kickoff authoring MUST be line-by-line matched to the committed SPEC.md. Memory-based authoring is prohibited. Every kickoff MUST:

1. **Cite SPEC section numbers explicitly** — "per SPEC §3 G2 steps 1-4" rather than "do the auth flow".
2. **Enumerate the full scope** — group count, success-criteria count, stop-point count all match SPEC exactly. No subsets unless the divergence is itself a CEO decision logged as a separate ADR amendment.
3. **Resolve every SPEC open question** (§6 style) — either accept proposal as-is or list the decision inline. Kickoff cannot assume defaults.
4. **Pre-flight check**: Builder's FIRST response to a kickoff is to spot-check kickoff ↔ SPEC/CEO_APPROVED alignment. Any discrepancy → Builder flags pre-execution; user retracts or amends the kickoff before Builder runs a single tool.

### Consequences
- Positive: scope drift caught at dictation time, not mid-build. +5 min per kickoff is a cheap trade vs a mid-phase surprise that may be irreversible (e.g., partial DB schema, landed migrations without matching code).
- Positive: Builder's pre-flight check habit hardens into a reliable gate (now explicitly mandated, not improvised).
- Negative: small overhead on every phase kickoff; may feel paperwork-heavy for trivial phases. Mitigation: for phases ≤3 groups and ≤2 success criteria, an inline SPEC reference in the kickoff body may stand in for section-by-section citation.

### Alternatives considered
- Strict-SPEC-only execution (no kickoff chat) — rejected: loses the CEO's real-time shaping / emphasis that chat prompts naturally carry.
- Post-execution reconciliation — rejected: divergence only surfaces after damage is done (landed commits, applied migrations).
- Format-driven (required template for kickoff messages) — deferred. If ADR-006 in practice doesn't catch drift, a template is the next escalation.

### Follow-ups
- If Phase 2.5 or 2.6 kickoff exhibits drift despite this ADR, escalate to a template in a follow-up ADR-007.

## ADR-009: Role type system reconciliation (Phase 1.1 ↔ Phase 2.5)

**Date**: 2026-04-23
**Status**: Accepted
**Context-trigger**: Phase 2.5 G2 entry (auth flow + role selection)

### Context

Phase 2.5 G1 introduced `profiles.role IN ('creator','studio','observer')`,
intended as the **persona role** for the public Challenge Platform. Per
Phase 2.5 SPEC v2 §1.2, this is **orthogonal** to the existing Phase 1.1
system `user_roles.role IN ('yagi_admin','workspace_admin','workspace_member','creator')`.

The literal string `creator` collides between the two systems:

- `user_roles.role='creator'` (Phase 1.1) — used in `src/app/[locale]/app/layout.tsx`
  as part of `hasPrivilegedGlobalRole` to allow workspace-less app access.
  Originally seeded to enable solo-creator usage of preprod / showcase tools.
- `profiles.role='creator'` (Phase 2.5) — the AI Creator persona who
  participates in Challenges. Has nothing to do with workspace permissions.

Without disambiguation, downstream code reading `roles.includes('creator')`
will silently mean different things in different files.

### Decision

Adopt **Option C: type discrimination + scope hook**.

#### Type system

In `src/lib/app/context.ts`:

```ts
// Phase 1.1 workspace permission system — unchanged literal, renamed type.
export type WorkspaceRole =
  | "yagi_admin"
  | "workspace_admin"
  | "workspace_member"
  | "creator";  // legacy: "workspace-less app access" semantic

// Phase 2.5 challenge persona system — distinct type, distinct namespace.
export type ProfileRole = "creator" | "studio" | "observer";

// AppContext exposes both as separate fields. Never overloaded.
export type AppContext = {
  userId: string;
  profile: {
    id: string;
    handle: string;
    display_name: string;
    avatar_url: string | null;
    locale: "ko" | "en";
    role: ProfileRole | null;  // NEW — from profiles.role (Phase 2.5)
  };
  workspaceRoles: WorkspaceRole[];  // RENAMED from `roles`
  workspaces: { id: string; name: string; slug: string }[];
  currentWorkspaceId: string | null;
};
```

Migration impact:
- `ctx.roles` → `ctx.workspaceRoles` (mechanical rename, all call sites)
- `ctx.profile.role` is new (Phase 2.5 G2 populates this from `profiles.role`)
- No DB change

#### Naming rule (durable)

Whenever a code site needs to ask "what kind of user is this?", **always
prefix with the system name**:
- `workspaceRoles.includes('creator')` — "is this user a Phase 1.1 workspace
  creator (workspace-less app access entitlement)?"
- `profile.role === 'creator'` — "is this user a Phase 2.5 AI Creator
  persona (Challenge participant)?"

Code reviewers reject any usage of bare `roles.includes('creator')` after
this ADR — it's a type error AND a semantic error.

### Consequences

- (+) Type system surfaces the conflict at every call site; impossible
      to write `roles.includes('creator')` without TypeScript error
      pointing at the namespace ambiguity.
- (+) Reads like documentation: `profile.role === 'creator'` is
      unambiguous in code review.
- (+) Phase 1.1 rows untouched. Migration is TS-only, instant.
- (–) Mechanical rename touches every file that reads `ctx.roles`. Likely
      ~10-15 sites based on grep. Trivial codemod.
- (–) `creators` table (Phase 2.5) and `creator` workspace role share
      stem. Mitigated by the strict prefix rule.

### Alternatives considered

- **Option A**: Rename Phase 1.1 `creator` to `solo_user` immediately.
  Rejected: requires DB migration on already-shipped Phase 1.1 system.
- **Option B**: Add `WorkspaceRole` / `ProfileRole` types but keep
  shared `roles` field. Rejected: shared field still invites bug class.
- **Option D**: String prefixes (`ws:creator`, `pr:creator`). Rejected:
  ugly at call site, requires runtime parsing.

### Followups (deferred)

- Phase 3+: Rename Phase 1.1 `creator` → `solo_user` once schema migration
  cost is acceptable. Logged in `.yagi-autobuild/phase-2-5/FOLLOWUPS.md` FU-9 candidate.

### Adoption (Builder at G2 entry)

1. Edit `src/lib/app/context.ts`: introduce `WorkspaceRole` + `ProfileRole`
   types, rename field `roles` → `workspaceRoles`, add `profile.role`.
2. Update `fetchAppContext`: select `role` from `profiles` table, return
   in `profile.role`.
3. Codemod: `git grep "ctx.roles" src/` → rename to `ctx.workspaceRoles`.
4. Codemod: `git grep "roles.includes('creator')" src/` → audit each site.
5. Run `tsc --noEmit` — should be clean after all rename sites updated.

### Cross-references

- `.yagi-autobuild/phase-2-5/G2-ENTRY-DECISION-PACKAGE.md` §A
- `.yagi-autobuild/phase-2-6/SPEC.md` §2 PRE-1

## ADR-010: Sidebar IA grouping + billing retirement + invoice consolidation
Date: 2026-04-24
Status: Accepted

### Context

Phase 2.5 added 3 public surfaces + 1 admin challenge console to the app.
The existing sidebar (Phase 1.x baseline) is a flat 11+ item list with
heterogeneous categories — past Miller 7±2, no visual grouping, and no
information hierarchy beyond workspace context. New routes (`/app/admin/challenges/*`) would push the list to 14+ items, degrading findability further.

Three latent issues surface at the same time:

1. **No grouping** — operations tier mixes work-output (projects, preprod,
   showcases), communication (meetings, notifications, team), money
   (invoices, admin_invoices), and system (settings, admin) without
   visual separation.
2. **Aspirational slot with no roadmap** — `billing` renders as a
   disabled "(Soon)" entry in Phase 1.x, but there is no Phase plan
   for self-serve usage billing and no ADR establishing what `billing`
   would even mean. It signals scope we don't own.
3. **Invoice surface split** — `invoices` (workspace-received) lives in
   the main list, `admin_invoices` (YAGI-issued) lives in a bottom
   admin divider. Both are the same money surface viewed from different
   sides; splitting them across the tier obscures the coherence.

### Decision

1. **Introduce 4-group operations tier.** Flat list becomes quiet
   uppercase sections: `작업` (work output) / `소통` (communication) /
   `결제` (money) / `시스템` (system). Labels auto-hide when a group
   has fewer than 2 visible items for the current user role, so the
   visual weight tracks role, not structure.
2. **Retire `billing`** from the sidebar. Entry removed from the items
   array and translation keys (`nav.billing`) removed. If SaaS usage
   billing becomes a real phase, it re-enters via a new ADR + mapping
   amendment.
3. **Consolidate invoices under 결제.** `invoices` + `admin_invoices`
   become two children of a single `결제 ▾` parent row. When only one
   is role-visible, the parent wrapper collapses into a bare link so
   workspace-only users see exactly one `결제` entry with no redundant
   group chrome.
4. **2-depth indent with auto-expand.** Parents with no `href` act as
   non-clickable toggles. Parent auto-expands when the current pathname
   matches any child route. Used by `챌린지 ▾` (yagi_admin — list + new
   + open-filter) and `결제 ▾` (yagi_admin — received + issued).

### Consequences

- (+) Operations tier drops from ~11 flat items to 4 labeled groups;
      each group stays well under Miller 7±2.
- (+) Role-dynamic auto-hide means non-admin users see a smaller sidebar
      without us authoring separate role trees — one config, runtime
      filter.
- (+) Invoice coherence: `결제 ▾` gathers both perspectives under one
      parent. Admin users see the money surface as a unit instead of
      two disconnected items.
- (+) `billing` removal eliminates the longest-standing "(Soon)" entry
      that had no roadmap, aligning with ADR-001-adjacent
      aspirational-slot-retirement principle from ARCHITECTURE.md.
- (–) 2-depth introduces a small parent-row toggle interaction users
      didn't have before. Mitigated by auto-expand on active route.
- (–) `challenges_open` child uses `?state=open` query variant, which
      requires a custom active-state matcher (pathname + search).
      Complexity isolated to `sidebar-nav.tsx`.

### Alternatives considered

- **Keep flat list, rename labels for clarity.** Rejected — doesn't
  solve Miller 7±2; just re-labels the same density problem.
- **Label-less dividers between mental groups.** Rejected — requires
  users to infer categories; eliminates the self-documenting benefit
  of named groups.
- **Collapsible groups with persisted state.** Deferred — state
  persistence (client storage + hydration) is out of Phase 2.6 scope.
  Auto-expand-on-active-route gives 90% of the benefit without the
  storage surface. Revisit in Phase 3+ if feedback demands.
- **Keep `billing` as a visible aspirational slot.** Rejected — signals
  scope we don't own and trains users to expect an unshipped feature.

### Followups

- **FU-SIDEBAR-STORYBOARD-1** (`.yagi-autobuild/phase-2-6/FOLLOWUPS.md`)
  — `storyboards` and `brands` remain disabled placeholders. Flip
  `disabled: false` when those phases ship.
- **FU-GUIDES-1** — remaining 3 help-route entries ship at
  `published: false` in Phase 2.6; content authoring is cross-phase.
- Growth triggers (SPEC §4) — any group >5 items, 3-depth warranted,
  or "I can't find X" from ≥2 distinct users fires an IA review. Each
  expansion requires a new ADR.

### Adoption

1. `src/components/app/sidebar-nav.tsx` — flat `items` array replaced
   by `GROUPS` array; added `filterItem` / `computeActiveKey` /
   `ParentRow` helpers.
2. `src/components/app/sidebar-group-label.tsx` — new quiet-uppercase
   label component.
3. `messages/ko.json` + `messages/en.json` — added `nav.groups.*`
   keys, `nav.challenges*` keys, `nav.notifications`, `nav.showcases`,
   `nav.billing_group`; removed `nav.billing`.
