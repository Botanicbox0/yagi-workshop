# YAGI Workshop — Autopilot Chain Protocol

> **Purpose:** Phases 1.2 → 1.9 run as a single autonomous chain. Yagi doesn't kick off each Phase manually — the completed Phase's Builder reads the next spec and transitions B-O-E automatically.
> **Boundary:** Kill-switches inside each Phase still require Telegram continue from Yagi. Autopilot eliminates idle time BETWEEN Phases, not oversight WITHIN Phases.
> **Design decision:** ARCHITECTURE.md §10.

---

## Phase completion protocol (runs at END of every Phase's Builder)

After all subtasks pass and `pnpm build` is clean:

### Step 1: Codex adversarial review

```
/codex:adversarial-review --base main --background <focus prompt from codex-review-protocol.md for this phase>
```

Poll with `/codex:status` until complete. Get `/codex:result`. Apply validation filter (see `codex-review-protocol.md` — "Validation pattern (noise filter)"). Fix surviving findings as a final Executor wave. Re-run `pnpm build`.

If any surviving findings remain unfixable within 30 min: Telegram with findings + ask Yagi `continue` (ship anyway) or `abort` (hold Phase).

### Step 2: Phase summary

Write `.yagi-autobuild/summary-phase-{N}.md` with:
- Subtask completion status (✅ / ⚠️ deviations / ❌ skipped)
- Any architectural decisions made (and append to ARCHITECTURE.md §14+ if novel)
- Codex review findings summary (kept vs filtered out, with rationale)
- Env vars required for the next Phase (if any)
- Known open issues

### Step 3: Telegram completion message

```
✅ Phase {N} complete

Subtasks: {passed} / {total}
Codex review: {findings_kept} kept, {findings_filtered} filtered
Duration: {hh:mm}

→ Autopilot: checking prereqs for Phase {N+1}...
```

### Step 4: Prereq check for next Phase

Lookup next Phase in ROADMAP.md. Read its `phase-{N+1}-spec.md` header for `Autopilot gate` line.

**Env var gates:**

| Phase | Required env vars |
|---|---|
| 1.3 | `GOOGLE_OAUTH_CLIENT_ID`, `GOOGLE_OAUTH_CLIENT_SECRET`, `GOOGLE_OAUTH_REFRESH_TOKEN` |
| 1.5 | `POPBILL_LINK_ID`, `POPBILL_SECRET_KEY`, `POPBILL_CORP_NUM` |

Check `.env.local` for presence (non-empty values). Do NOT attempt to validate the values themselves — just presence.

**If all required env vars present:** proceed to Step 5.

**If any missing:** halt the chain with a Telegram notice:

```
🛑 Autopilot halted — Phase {N+1} requires env vars.

Missing:
- GOOGLE_OAUTH_CLIENT_ID
- GOOGLE_OAUTH_REFRESH_TOKEN

Setup runbook: docs/google-oauth-setup.md (will be created by Phase {N+1} spec's subtask 02 if not present)

When ready, reply to this message with `continue phase {N+1}` to resume.
```

Then wait. No transition. Yagi handles env setup and manually kicks off the next Phase when ready.

### Step 5: Spec validation

Read `phase-{N+1}-spec.md` and `task_plan-{N+1}.md` (if exists). Verify:
- Spec has a "Goal" section
- Spec has a "Subtasks" section with numbered tasks
- Spec has a "Success criteria" section
- Spec has a "Model routing" section

If any section missing: halt with Telegram `🛑 Phase {N+1} spec malformed — missing "<section>" section.`

### Step 6: Fresh context transition

B-O-E cannot carry context from one Phase to the next (token budget + confusion). The transition uses a **fresh context bootstrap**:

Builder writes to `.yagi-autobuild/kickoff-{N+1}.md`:

```
# Phase {N+1} Kickoff — Auto-generated by Autopilot

Previous phase: {N} ({completion_timestamp})

## Boot sequence

1. Read `/CLAUDE.md`
2. Read `/ARCHITECTURE.md` sections mentioned in `phase-{N+1}-spec.md` header
3. Read `.yagi-autobuild/phase-{N+1}-spec.md` fully
4. Read `.yagi-autobuild/codex-review-protocol.md` — "Per-phase adversarial focus areas"
5. Read last 3 `summary-phase-*.md` for context on recent deviations
6. Start B-O-E per spec's Parallelism plan

## Operator note

This is an Autopilot transition — Yagi is not expected to be watching in real-time.
Kill-switches in this Phase still halt for Telegram continue.

## Spec summary for quick orient

{copy from phase-{N+1}-spec.md Goal + Non-goals sections}
```

Then Builder:
1. `/compact` the current Claude Code session
2. Spawn the Phase {N+1} B-O-E: paste the kickoff.md contents as the first message in a fresh Builder prompt
3. This is handled by the same Claude Code session — we don't spin up a separate instance

### Step 7: Handoff complete

Builder's final message in its own context window:

```
✅ Autopilot handoff complete.
Phase {N+1} Builder now active with fresh context.
Monitor via Telegram for kill-switches.
```

## Special handling: Phase 1.2 → 1.2.5 (first transition)

Phase 1.2 was kicked off manually (before Autopilot existed). Its Builder doesn't have Autopilot instructions baked in.

**One-time patch for Phase 1.2's completion:**

When Phase 1.2 Builder sends its `✅ Phase 1.2 complete` Telegram, Yagi replies:

```
continue autopilot: from now on, follow .yagi-autobuild/AUTOPILOT.md protocol for all phase transitions. Start with Phase 1.2.5 per ROADMAP.md.
```

The Builder then:
1. Reads this file (AUTOPILOT.md)
2. Reads `codex-review-protocol.md`
3. Runs Codex review retroactively on Phase 1.2 (Step 1)
4. Generates Phase 1.2 summary (Step 2 — retroactive)
5. Proceeds to Step 4 onward for transition to 1.2.5

From Phase 1.2.5 onward, Autopilot is baked into every Phase spec's "Final Builder actions" section.

## Failure modes

### Kill-switch abort mid-Phase

If Yagi replies `abort` to any kill-switch during Phase N: the chain halts immediately. No transition to N+1. Builder writes a partial summary documenting where Phase N stopped.

### Codex review finds critical issue

If Codex review finds a concrete exploitable issue (e.g., RLS bypass) that Builder cannot fix within 30 min: halt the chain, post to Telegram with the finding. Yagi decides whether to ship the Phase with a known issue (tracking it) or hold.

### Build fails after Codex fixes applied

If `pnpm build` breaks after applying Codex-suggested fixes: Builder reverts the last set of changes, re-runs build, documents the "Codex suggested but build-breaking" finding in the summary, Telegrams Yagi for a decision.

### External API outage during Phase

If a Phase depends on an external service (fal.ai was one such case — removed in 1.4 redesign; Google Calendar for 1.3, 팝빌 for 1.5, Resend for 1.8) and that service is down at kickoff or during testing: Builder kicks off the Phase anyway, implements the escape hatch path (per ARCHITECTURE.md §1.5), verifies escape-hatch behavior, defers the happy-path smoke test to when the service returns. Telegram notes the degraded state.

### Autopilot chain interrupted by Claude Code session expiration

If the Claude Code session is closed (manually or by timeout) mid-chain: the next time Yagi opens Claude Code in this project, the first command should be:

```
read .yagi-autobuild/AUTOPILOT.md and check .yagi-autobuild/summary-*.md to find the most recent completed phase; resume the chain from the next phase
```

This bootstraps a fresh session back into the chain state. The Phase specs are self-contained enough that a fresh Builder can resume without Yagi re-explaining context.

## Escape switches

Any of these in a Phase kickoff or kill-switch reply aborts Autopilot and puts the chain on hold:

- `abort`
- `stop autopilot`
- `manual mode`
- `--no-autopilot` (flag when Yagi kicks off a Phase manually)

Yagi can also disable Autopilot for the next Phase only:

```
continue, but --no-autopilot after this phase
```

The current Phase completes normally, but no transition to the next Phase — Yagi kicks off the next manually.

## Observability

Every Autopilot action is logged to `.yagi-autobuild/autopilot-log.md` (append-only):

```
2026-04-22 03:14 · Phase 1.2 completed (Codex clean)
2026-04-22 03:15 · Autopilot transition → Phase 1.2.5
2026-04-22 03:15 · Prereq check passed (no env vars required)
2026-04-22 03:16 · B-O-E spawned for 1.2.5
2026-04-22 05:42 · Phase 1.2.5 kill-switch 1 → Telegram sent
2026-04-22 05:45 · Yagi replied `continue`
...
```

This log is the single source of truth for what Autopilot did while Yagi was away.

## Forbidden

- Running two Phases in parallel. Strict serial execution only.
- Skipping the Codex review step "to save time." The dual-model review is non-negotiable per ARCHITECTURE §8.6.
- Auto-overriding an `abort` with a `continue` if Yagi changes their mind via chat. Every kill-switch requires an explicit new reply.
- Auto-writing env vars to `.env.local`. If vars are missing, halt — never guess values.
- Consuming more than 10 minutes of token budget on a single Codex review filter pass. If it's taking longer, ship the Phase with the raw Codex output attached to the summary and move on.
- Auto-publishing Phase 1.9 showcases on behalf of a client without the "I have approval to publish" checkbox from YAGI admin.

---

## Summary

Autopilot = "Yagi goes to sleep after Phase 1.2 is building; wakes up to Phase 1.9 shipped (or a specific Telegram reason why it's halted)."

Human oversight remains intact via kill-switches. Autopilot only eliminates the "manual kickoff" idle time between Phases.
