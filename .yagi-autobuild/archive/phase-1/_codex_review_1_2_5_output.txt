[HIGH] src/lib/thread-attachments.ts:10 / src/components/project/thread-panel.tsx:301 — Internal attachment storage access is weaker than message visibility
Exploit: The attachment row RLS may hide internal rows, but any non-admin workspace member who learns an internal `storage_path` can still call the same browser-side `createSignedUrl()` flow and fetch the object because storage policy only keys on the first path segment.
Fix: Mirror internal-message visibility in `storage.objects` policy for `thread-attachments`, or move signing behind a server action that re-checks `thread_messages.visibility` before issuing a URL.

[MEDIUM] src/lib/og-video-unfurl.ts:121 — oEmbed fetch follows redirects without per-hop SSRF revalidation
Exploit: A trusted oEmbed endpoint that 302s to an internal host/IP will be followed by `fetch(..., { redirect: "follow" })` even though only the initial URL passed `validateHost()`.
Fix: Replace `redirect: "follow"` with the same manual redirect walker used in `src/lib/og-unfurl.ts`, validating every hop before fetching it.

[MEDIUM] src/components/project/thread-panel.tsx:146 / src/components/project/thread-panel.tsx:392 — Other clients never hydrate attachments for realtime-inserted messages
Exploit: Collaborators who did not send the message receive the new `thread_messages` row over realtime with no attachment payload, and it stays attachment-empty until a full reload.
Fix: On every realtime insert, fetch the full message payload including attachments, or subscribe to `thread_message_attachments` inserts and merge them client-side.

[LOW] src/components/project/thread-panel-server.tsx:6 / src/components/project/thread-panel.tsx:793 — 1-hour attachment video URLs expire with no recovery path
Exploit: If a large/private video needs a new range request after the 3600s signed URL expires, the thread attachment player just stalls/fails because this render path has no retry or visible error handling.
Fix: Renew signed URLs on `error`/`stalled`, or serve videos through a refreshable redirect endpoint instead of embedding fixed one-hour URLs.

[LOW] src/lib/references/pdf.ts:15 — PDF worker is executed from a third-party CDN with no origin pinning
Exploit: A compromised jsDelivr worker can run arbitrary worker code over the user’s PDF bytes and exfiltrate that document, even though it does not get direct access to `localStorage` or `document.cookie`.
Fix: Self-host the `pdfjs-dist` worker from your own origin or bundle it locally instead of loading executable code from a CDN at runtime.

[LOW] src/app/[locale]/app/projects/[id]/ref-actions.ts:56 — `addReference` trusts caller-supplied `media_type` and related metadata
Exploit: A client can upload one file type and persist a mismatched `media_type`/`embed_provider`/`page_count`, leaving permanently inconsistent reference rows that the UI renders incorrectly.
Fix: Derive `media_type` and its metadata from trusted upload/object metadata on the server, or re-read the uploaded object before insert and reject mismatches.
