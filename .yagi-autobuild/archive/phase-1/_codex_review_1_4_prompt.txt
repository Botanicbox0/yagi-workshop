SPECIFIC ADVERSARIAL FOCUS (rank findings HIGH/MEDIUM/LOW; cite file:line; one-line fix each):

1. **Public share endpoint security** — /api/share/[token]/* and /s/[token]
   (a) Rate limit is in-memory Map per serverless instance. Can an attacker spawn enough cold starts to bypass the limit? What is the actual blast radius (e.g. how many emails could a bad actor send via the comments endpoint before getting throttled)?
   (b) Token enumeration: tokens are 32-byte base64url. Confirm. Any path where the API leaks "token exists but board is archived" vs "token doesn't exist" allowing enumeration?
   (c) UPSERT on (frame_id, reactor_email) — can a malicious client poison another reactor's vote by submitting a reaction with someone else's email? (Yes by design, but verify the surface.)
   (d) Comment body — escaped before insert? Stored XSS surface when YAGI views it in the editor or in the Resend notification email?
   (e) approve endpoint sets approved_by_email from request body — any verification, or can anyone "approve" any shared board? If anyone can approve, is that the intended UX or a security flaw given that approval transitions board status irreversibly?
   (f) Service-role client in /s/[token] — does any error path leak service-role-only data (e.g. internal IDs, other workspaces' data) back to the public response?

2. **Workspace_id trigger correctness** — preprod_boards_set_workspace_id_ins
   (a) Trigger is SECURITY DEFINER. Confirm it's owned by a role with no superpowers it shouldn't have. Could trigger be bypassed via INSERT ... ON CONFLICT DO UPDATE (the trigger fires only on plain INSERT/UPDATE)?
   (b) Can a non-yagi_admin caller use the createBoard server action to insert a board for a workspace they're not a member of, IF the trigger derivation fails (e.g. yagi-internal workspace was renamed/deleted)?
   (c) What if the trigger BLOCKs (yagi-internal not found) — does it raise NOTICE, NULL the workspace_id, or RAISE EXCEPTION? An incomplete write surface could leave half-applied state.

3. **RLS bypass on preprod tables**
   (a) preprod_boards_select policy: is_yagi_admin(auth.uid()) OR is_ws_member(auth.uid(), workspace_id). Since the workspace_id is *yagi-internal*, can a non-yagi-admin who happens to be a workspace_member of yagi-internal see ALL boards across all client projects? Is this intentional? If so, who should be granted yagi-internal membership?
   (b) preprod_frames_select inherits via EXISTS on board. Frame UPDATE policy — can a ws_admin of yagi-internal change frame's board_id to a different board (potentially leaking media into another board)?
   (c) preprod_frame_reactions / preprod_frame_comments — public clients submit via service-role API routes. RLS only protects against direct PostgREST. Confirm there's no Anon JWT path that could write directly bypassing rate limit.
   (d) Storage bucket preprod-frames is private. RLS keyed on (storage.foldername(name))[1] = board.id::text. What if an attacker supplies a path like `someBoardId/../otherBoardId/file.png`? Does Supabase storage normalize the path, or could traversal escape the policy?

4. **Versioning correctness — createFrameRevision / restoreFrameRevision**
   (a) Race: two concurrent createFrameRevision calls on the same frame. Does the conditional UPDATE-then-INSERT pattern ensure only one wins, or can both succeed leaving two is_current_revision=true rows? (Unique partial index on revision_group where is_current_revision=true should prevent — verify the index exists in migration.)
   (b) Rollback path on insert failure: re-promotes the demoted row. What if the rollback UPDATE itself fails? State is left with NO current revision — frame disappears from share page.
   (c) restoreFrameRevision copies historical row as new revision+1 — does it copy media_storage_path? If the old storage object was deleted (cleanup), the restored revision is broken.

5. **Realtime subscription correctness** — board-editor.tsx
   (a) Subscription filter `board_id=eq.${board.id}` — string interpolation of board.id (UUID). Could a malicious board.id (if ever crafted client-side) inject into the filter? In practice board.id comes from the server-side RSC, not user input. Confirm.
   (b) Cleanup on unmount uses supabase.removeChannel(channel). If the user navigates away mid-subscription handshake, is there a leaked WS connection?
   (c) Realtime requires RLS-replication permissions for anon/authenticated to receive events. Did the migration grant SUPABASE_REALTIME or alter publication for these tables? If not, subscriptions silently no-op — feature broken.

6. **Email leakage via notifications** — comments + approve endpoints send Resend email to EMAIL_FROM
   (a) Email body — does it interpolate the comment body / reactor name unescaped into HTML? Stored XSS in YAGI's inbox?
   (b) Does any error path log the resend API key, service-role key, or NEXT_PUBLIC_SITE_URL with sensitive query params?
   (c) Reactor identity is provided by the client (not verified). Could be used to spoof "feedback from CEO@client.com" in YAGI's inbox. Document if intentional.
